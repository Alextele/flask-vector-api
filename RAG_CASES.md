# RAG System — Кейсы

Эволюция от простого LLM-чата до полноценной RAG-системы.

---

## Кейс 1 — Bare LLM Chat

**Что это:** прямое обращение к языковой модели (LLM) без какого-либо контекста.

- Модель отвечает на основе своих весов, полученных при предобучении (pre-training knowledge)
- Никакой привязки к домену нет — модель может галлюцинировать, отвечать на любые вопросы
- В проекте: `POST /generate` с только `question` и `system_prompt`

**Ограничение:** модель не знает ничего о твоём бизнесе/продукте, её поведение непредсказуемо.

---

## Кейс 2 — Few-Shot Prompting / Context Injection

**Что это:** ограничение поведения модели через системный промпт и заранее заготовленные примеры вопрос-ответ (few-shot examples).

- `system_prompt` задаёт роль и границы поведения модели ("Ты — ассистент магазина X, отвечай только про...")
- В поле `context` передаются примеры или правила как часть промпта
- Техника называется **in-context learning** — модель адаптируется к задаче без дообучения (fine-tuning)
- Вариант без векторного поиска: контекст подкладывается статично, вручную

**Ограничение:** контекст ограничен размером окна (`n_ctx`). При большой базе знаний всё не влезет.

---

## Кейс 3 — RAG: Semantic Search + Generation

**Это классический RAG (Retrieval-Augmented Generation).**

Пайплайн состоит из двух этапов:

### Retrieval (поиск)
- Вопрос пользователя преобразуется в вектор через Embedding-модель (`POST /vector` → `cointegrated/rubert-tiny`)
- В векторной базе данных (Chroma, Qdrant, FAISS, Weaviate) хранятся эмбеддинги одобренных вопросов/документов
- Вычисляется **cosine similarity** (или L2-расстояние) между вектором вопроса и векторами базы
- Если расстояние превышает порог (**similarity threshold**) — вопрос считается **out-of-domain**: модель отвечает отказом ("Это не входит в мою компетенцию")
- Если вопрос релевантен — извлекаются топ-K наиболее похожих документов/ответов (**top-K retrieval**)

### Augmentation + Generation
- Найденные документы подкладываются в `context` как **grounding** — земля под ногами для модели
- LLM (`saiga_yandexgpt_8b`) генерирует ответ, опираясь на этот контекст, а не на параметрическую память
- Это называется **grounded generation** — ответ заземлён на реальные данные

**Ключевое свойство RAG:** база знаний обновляется независимо от модели. Не нужен fine-tuning — достаточно добавить новые документы в векторную БД.

## Кейс 4 — Conversational RAG (Multi-Turn Dialogue)

**Что это:** добавление памяти диалога (**conversation history / dialogue state**).

- История хранится как список сообщений `[{"role": "user", "content": ...}, {"role": "assistant", "content": ...}]`
- При каждом новом вопросе история передаётся в LLM, что позволяет модели понимать **кореференции** ("а что насчёт второго варианта?" — модель понимает, о чём речь)
- Для retrieval история тоже важна: техника **query rewriting** — переформулировать текущий вопрос с учётом контекста диалога перед поиском в векторной БД
- Хранение истории: in-memory (для сессии), Redis (персистентно), или БД

**Проблема:** чем длиннее история, тем больше токенов занимает промпт. Решается через **conversation summarization** — сжатие старых сообщений в краткое резюме.

## Общая схема эволюции

```
Кейс 1        Кейс 2           Кейс 3                    Кейс 4
  │              │                │                           │
LLM only  →  Prompt       →   Vector DB +           →   History +
             Engineering       Semantic Search            Multi-turn
                                 + Grounding              RAG
```
